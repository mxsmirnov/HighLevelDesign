# Реестр корпоративных сервисов (Enterprise Service Catalog)

**Назначение:** Данный документ описывает доступные корпоративные информационные системы и их высокоуровневые API-возможности. Документ предназначен для архитектурного проектирования: на его основе проектируются сценарии интеграции и выбираются источники данных для новых решений.

**Версия схемы:** 1.0
**Последнее обновление:** 2026-03-11

---

## 1. Система управления взаимоотношениями с клиентами (CRM)
*Домен: Продажи и поддержка*

*   **Система:** Salesforce / MS Dynamics / custom CRM "ClientNet"
*   **Роль в ландшафте:** Основной источник данных о клиентах (юридических и физических лицах), контрагентах, сделках и контактных лицах.
*   **Способы доступа:**
    *   **REST API:** `https://api.crm.company.com/v2` (Документация: [ссылка на Confluence/Swagger])
    *   **GraphQL Gateway:** `https://graphql.company.com/gateway/crm` (Ограниченный сабграф)
    *   **SFTP (для выгрузок):** `sftp://data-lake.company.com/inbound/crm` (для пакетной обработки > 1 млн записей)
*   **Бизнес-объекты (ключевые):**
    *   `Account` (Контрагент): Идентификатор, наименование, ИНН, статус, сегмент.
    *   `Contact` (Контактное лицо): ФИО, email, телефон, роль, ссылка на `Account`.
    *   `Opportunity` (Сделка): Сумма, стадия, дата закрытия, менеджер.
*   **Ограничения:**
    *   Rate Limit: 1000 запросов в минуту на приложение.
    *   Исторические данные хранятся 3 года.
    *   Поиск по ИНН требует точного совпадения (full-text поиск отключен).

## 2. ERP (Управление ресурсами предприятия)
*Домен: Финансы и логистика*

*   **Система:** SAP ECC / 1С:ERP / Oracle
*   **Роль в ландшафте:** Система учета заказов, управления запасами, финансовых проводок и расчета себестоимости.
*   **Способы доступа:**
    *   **REST API (SAP Gateway):** `https://sap-erp.company.com/sap/opu/odata` (OData v4)
    *   **SOAP Web Services:** `http://esb.company.com/legacy/erp?wsdl` (Устаревший, но поддерживаемый канал для критических операций)
    *   **Kafka (CDC):** Топик `raw.cdc.erp.materials` (стриминг изменений по справочнику материалов).
*   **Бизнес-объекты (ключевые):**
    *   `SalesOrder` (Заказ клиента): Номер, дата, статус отгрузки, сумма, ссылка на `Account` (из CRM) - *Внимание: маппинг ID клиента происходит через таблицу трансляции в ESB.*
    *   `Material` (Номенклатура): Артикул, наименование, цена, единица измерения, тип (услуга/товар).
    *   `Invoice` (Счет-фактура): Номер, сумма, ссылка на `SalesOrder`.
*   **Ограничения:**
    *   OData сервисы работают с задержкой до 2 секунд (не использовать для синхронного UI).
    *   Изменения в `SalesOrder` отражаются в Kafka с задержкой ~15 минут.

## 3. Система документооборота (ECM)
*Домен: Юридические и кадровые документы*

*   **Система:** Directum / DocuShare / custom "DocFlow"
*   **Роль в ландшафте:** Хранилище скан-копий договоров, актов, первичной документации.
*   **Способы доступа:**
    *   **REST API (только чтение):** `https://doc-api.company.com/v1` (Swagger: [ссылка])
    *   **Webhook (уведомления):** Система может стучаться в `{your-service}/webhook/docflow/status` при смене статуса документа (требуется предварительная настройка).
*   **Бизнес-объекты (ключевые):**
    *   `Contract` (Договор): Уникальный ID, статус (Черновик/Подписан/Расторгнут), ссылка на файл, дата подписания.
    *   `DocumentPackage` (Пакет документов): Набор связанных `Contract` и доп. соглашений.
*   **Ограничения:**
    *   Загрузка файлов через API запрещена (только через веб-интерфейс).
    *   Поиск работает только по метаданным, OCR для скан-копий не поддерживается.

## 4. Шина данных и потоковая обработка (ESB / Kafka)
*Домен: Интеграционная шина*

*   **Система:** Confluent Kafka / RabbitMQ / MuleSoft
*   **Роль в ландшафте:** Транспорт для событий между системами. Позволяет реагировать на изменения данных без прямого опроса баз данных.
*   **Способы доступа:**
    *   **Схема (Schema Registry):** `https://schema-registry.company.com` (Avro/Protobuf схемы для всех топиков)
    *   **Топики для подписки (Consumer):**
        *   `domain.customer.updated` — Событие изменения данных клиента в CRM.
        *   `domain.order.created` — Событие создания нового заказа в ERP.
        *   `domain.payment.received` — Событие получения оплаты (из банк-клиента).
    *   **Топики для публикации (Producer):**
        *   `app.new-service.notification` — Сюда новый сервис может писать свои события (после согласования схемы).

## 5. Хранилище данных (DWH)
*Домен: Аналитика и отчетность*

*   **Система:** Greenplum / Vertica / Snowflake
*   **Роль в ландшафте:** Витрины данных для аналитики. Содержат очищенные, согласованные и агрегированные данные из всех систем.
*   **Способы доступа:**
    *   **JDBC/ODBC:** `jdbc:postgresql://dwh.company.com:5432/analytics` (только чтение)
    *   **REST API (SQL over HTTP):** `https://sql-api.company.com/v1/query` (Для простых запросов из микросервисов, только SELECT).
*   **Схемы данных:**
    *   `datamart.sales.daily` — Витрина ежедневных продаж (агрегат).
    *   `ods.crm.accounts` — Сырые данные по контрагентам (копия из CRM).

## Правила взаимодействия для новых проектов

1.  **Выбор источника:** Для получения *текущего состояния* объекта (например, «дать данные клиента») использовать синхронный REST API системы-источника (CRM/ERP). Для отслеживания *изменений* и реактивной архитектуры использовать Kafka.
2.  **Маппинг идентификаторов:** При связывании объектов из разных систем (например, Заказ из ERP с Клиентом из CRM) всегда использовать сервис трансляции `entity-lookup.service.company.com`.
3.  **Трассировка:** Все HTTP-запросы к системам должны содержать заголовок `X-Request-ID` для сквозной трассировки.
4.  **SLA:** Время критичных синхронных вызовов не должно превышать 500 мс. Если интеграция требует цепочки вызовов (A->Б->В), проектировать асинхронный сценарий через Kafka.
