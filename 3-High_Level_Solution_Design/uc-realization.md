
## Информационный запрос к aiExpert
ID: UC-01
**Название:** Обработка пользовательского запроса с использованием RAG.
**Акторы:** Пользователь (человек).
**Цель:** Получить точный, обоснованный ответ на вопрос, основанный на базе знаний агента, с возможностью уточнения диалога.
**Уровень:** Задача пользователя (User-Goal).

### Основной сценарий (Main Success Scenario):

1. <u>Пользователь</u> отправляет текстовый запрос в систему.
2. Агент <u>aiExpert</u>: Инициирует граф обработки, передавая запрос узлу Query Processor.
3. <u>Query Processor</u>: Очищает запрос от шума (артефактов) и определяет намерение пользователя _(Intent Recognition)_. При необходимости, активирует контекст предыдущих сообщений _(Memory Skill)_.
4. <u>Query Rewriter</u> (Узел: Генератор Поисковых Запросов): Анализирует запрос и генерирует 1-3 гипотетических поисковых запроса для улучшения поиска (HyDE(Hypothetical Document Embeddings) или мультизапросность).
5. Параллельное выполнение (Fork): Система активирует два навыка поиска:
5.1 <u>Vector Search Skill</u>(Навык: Векторный Поиск): Ищет семантические соответствия в векторной БД.
5.2 <u>FTS Skill</u>(Навык: Полнотекстовый Поиск): Ищет точные совпадения ключевых слов в индексе.
6. <u>Reranking & Fusion</u>(Узел «Ранжирование и Слияние»): Собирает результаты от обоих навыков, дедуплицирует их и применяет алгоритм ранжирования (например, Reciprocal Rank Fusion или модель реранкера типа Cross-Encoder) для выбора топ-K наиболее релевантных чанков (фрагментов текста).
7. <u> Prompt Assembler</u>(Узел «Формирование Контекста»): Оборачивает отобранные чанки в промпт, добавляя инструкции для LLM и историю диалога.
8. <u>LLM Node</u>(Узел «Генерация Ответа»): Отправляет контекст в LLM.
8.1 <u>Обработка ошибок</u> (Try-Success): Если LLM отвечает корректно, ответ парсится.
8.2 <u>Output Guardrails</u> (ВалидацияОтвета): Проверяет ответ на наличие галлюцинаций (Hallucination Check) относительно предоставленного контекста.
9. Агент <u>aiExpert</u> возвращает Пользователю структурированный ответ (текст + ссылки на источники).
10. <u>Memory Skill</u>(Узел: «Запись в Память» ): Сохраняет итоговый запрос и ответ в краткосрочную память диалога для использования в будущих шагах (шаг 3).

### Расширения (Extensions):

#### 3a. Неопределенное намерение:
1. Агент запускает уточняющий диалог (Clarification Loop).
2. Пользователь уточняет запрос.
3. Возврат к шагу 3 основного сценария.

#### 6a. Низкое качество или отсутствие результатов поиска:
1. Узел ранжирования возвращает пустой или низкокачественный набор документов (ниже порога релевантности).
2. Агент переходит в режим «Ответ на основе общей информации» (Fallback).
3. Узел генерации формирует ответ, начинающийся с фразы: "Я не нашел точной информации в базе знаний, но могу предположить...".
4. Сценарий продолжается с шага 7 (с пустым контекстом) или прерывается с выдачей сообщения об отсутствии данных.

#### 8a. Ошибка вызова LLM (Timeout / API Error):
Система ловит исключение (Try-Catch в узле). Навык Обработки Ошибок (Error Handling Skill): Активируется логика повторных попыток (retry) с экспоненциальной задержкой (до 3 раз).
* 8a1. Успех после повторов: Сценарий продолжается с шага 8 (Генерация).
* 8a2. Все попытки исчерпаны:
Агент возвращает пользователю дружелюбное сообщение об ошибке: «Сервис временно недоступен, повторите позже».
Записывает инцидент в лог для мониторинга (Observability).

#### 8b. Обнаружена галлюцинация (сработал Guardrails на шаге 8):
Ответ отклоняется.
Агент пытается перегенерировать ответ с более строгими инструкциями (повтор шага 7-8, один раз).
Если повторная генерация не помогла — возвращается сообщение о невозможности дать ответ на основе имеющихся данных.

### Техническая реализация на LangGraph (Technical Notes для разработчика):
* Граф (Graph): Циклический граф с условными ребрами (Conditional Edges).
* Состояние (State): Типизированный словарь (TypedDict), содержащий messages, retrieved_docs, search_queries, error_count.
* Ноды (Nodes):

**query_processor**: Использует небольшой LLM или эвристики.
**search_router**: Распараллеливает вызовы инструментов (Parallel node execution).
**retrieval_node**: Инструмент (ToolNode) для вызова векторного поиска (FAISS/Pinecone) и FTS (Elasticsearch/Postgres TSV).
**grade_documents**: Узел с LLM для оценки релевантности (или реранкер).
**generate**: Чат-модель с подгрузкой промпта.
**error_handler**: Узел, проверяющий error_count в состоянии и возвращающий поток либо на повтор, либо на финальный ответ с ошибкой.
